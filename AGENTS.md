# AGENTS.md

이 파일은 장소 기반 다이어리 서비스 프로젝트에서 AI 에이전트가 따라야 할 작업 규칙을 정의합니다.

## Project Overview

- 프로젝트 이름: 장소 기반 다이어리 서비스
- 프로젝트 목적: 사용자가 방문한 장소를 중심으로 다이어리를 작성하고, 지도 위에서 기록을 확인하며, 함께 가보고 싶은 장소 목록을 다른 사용자와 공동 관리할 수 있는 MVP를 만든다.
- 주요 사용자: 장소 중심으로 일상을 기록하고, 친구나 지인과 가보고 싶은 장소를 함께 정리하고 싶은 사용자
- 핵심 기능:
  - 방문 장소 기반 다이어리 CRUD
  - 작성된 다이어리의 지도 마커 표시
  - 가보고 싶은 장소 CRUD
  - 사용자 정의 카테고리 기반 장소 분류
  - 초대 코드 기반 공유 방 생성 및 입장
  - 공유 방 내 다이어리 조회
  - 공유 방 내 장소 목록 공동 편집

## Tech Stack

- Frontend: Vue, TypeScript
- Backend: Node.js, Express
- Database: PostgreSQL + PostGIS
- Map API: 카카오 지도 API
- Deployment: 미정. MVP 구현 이후 배포 범위를 결정한다.
- Package Manager: pnpm

## Working Rules

- 작업 전 확인할 문서:
  - `docs/PROJECT_PLAN.md`
  - 관련 GitHub Issue
  - 기능과 연결된 설계 문서, API 문서, 트러블슈팅 문서가 있다면 함께 확인한다.
- 작업 단위:
  - 기능 단위 GitHub Issue를 기준으로 작업한다.
  - Issue에는 기능 설명, 요구사항, 완료 조건, 테스트 기준, 관련 화면 또는 API를 포함해야 한다.
  - 큰 기능은 다이어리, 장소 목록, 공유 방, 지도 기능 등 작은 구현 단위로 나눈다.
- 작성 언어 규칙:
  - Issue 제목과 본문, PR 제목과 설명, 문서, 완료 보고는 가능한 한 한국어로 작성한다.
  - 기술 명사, 라이브러리명, 프레임워크명, API명, 파일명, 명령어는 영어 원문을 사용해도 된다.
- 브랜치 규칙:
  - 기본 브랜치에서 직접 기능 개발하지 않는다.
  - 기능 작업은 Issue 단위 브랜치에서 진행한다.
  - 브랜치명은 가능하면 `feature/issue-번호-요약`, `fix/issue-번호-요약` 형식을 사용한다.
- 커밋 규칙:
  - 하나의 커밋은 하나의 의도를 갖도록 작게 작성한다.
  - 커밋 메시지는 Conventional Commits 형식을 사용한다.
  - 커밋 메시지의 type은 `feat`, `fix`, `docs`, `chore`, `refactor`, `test`처럼 영어 키워드를 사용한다.
  - type 뒤 설명은 한국어로 작성해도 된다.
  - 예: `feat: 다이어리 생성 API 추가`, `fix: 초대 코드 검증 오류 수정`
- PR 규칙:
  - PR에는 작업한 Issue, 구현 내용, 테스트 결과, 남은 이슈를 포함한다.
  - PR 제목 앞에는 `[feat]`, `[fix]`, `[docs]`, `[chore]`, `[refactor]`, `[test]`처럼 작업 유형을 나타내는 접두어를 붙인다.
  - PR 제목에는 `[codex]`처럼 작성 주체를 나타내는 접두어를 붙이지 않는다.
  - MVP 범위를 벗어나는 변경은 별도 Issue로 분리한다.
  - 최종 머지는 사람이 직접 수행한다. AI 에이전트는 임의로 PR을 머지하지 않는다.
- 코드 리뷰 기준:
  - 요구사항과 완료 조건을 충족하는지 확인한다.
  - 유지보수 가능한 구조인지 확인한다.
  - 테스트 가능성과 예외 처리를 확인한다.
  - 공유 기능, 초대 코드, 사용자 데이터와 관련된 보안 이슈를 우선 검토한다.

## Coding Conventions

- 언어 및 프레임워크별 스타일:
  - TypeScript를 기본으로 사용한다.
  - Vue 컴포넌트와 Express API는 프로젝트 내부에서 정한 구조와 네이밍을 따른다.
  - 새 패턴을 도입하기 전 기존 코드 구조를 먼저 확인한다.
- 파일 및 폴더 네이밍:
  - 파일명과 폴더명은 기능과 책임이 드러나게 작성한다.
  - 동일 도메인의 파일은 한 영역에 모아 관리한다.
  - 도메인 기준은 사용자, 다이어리, 장소, 카테고리, 공유 방을 우선한다.
- 컴포넌트 작성 규칙:
  - 화면 단위 컴포넌트와 재사용 UI 컴포넌트를 구분한다.
  - 지도, 다이어리 폼, 장소 목록, 공유 방 정보처럼 역할이 분명한 단위로 분리한다.
  - 컴포넌트는 표시 책임과 데이터 처리 책임을 과하게 섞지 않는다.
- API 작성 규칙:
  - RESTful한 리소스 중심 API를 우선한다.
  - 요청 값 검증, 실패 응답, 권한 또는 참여 여부 검사를 명확히 처리한다.
  - 주요 리소스는 사용자, 다이어리, 장소, 카테고리, 공유 방, 초대 코드이다.
- 에러 처리 규칙:
  - 사용자 입력 오류, 존재하지 않는 리소스, 권한/참여 조건 불충족, 서버 오류를 구분한다.
  - 프론트엔드는 사용자가 다음 행동을 판단할 수 있는 메시지를 제공한다.
  - 백엔드는 디버깅 가능한 로그를 남기되 민감한 정보는 노출하지 않는다.
- 주석 작성 기준:
  - 코드만으로 의도가 분명하지 않은 정책, 예외 처리, 복잡한 흐름에만 주석을 작성한다.
  - 단순히 코드 동작을 반복 설명하는 주석은 작성하지 않는다.

## Testing Rules

- 테스트 실행 명령어:
  - Frontend: `cd fe && pnpm typecheck`
  - Frontend: `cd fe && pnpm build`
  - Frontend: `cd fe && pnpm test`
  - Backend: `cd be && pnpm typecheck`
  - Backend: `cd be && pnpm build`
  - Backend: `cd be && pnpm test`
  - Database: `docker compose config`
- 필수 테스트 범위:
  - 다이어리 CRUD
  - 장소 CRUD
  - 카테고리별 장소 분류
  - 초대 코드 생성 및 공유 방 입장
  - 공유 방 내 장소 목록 공동 편집
  - 주요 API 요청 검증과 실패 케이스
- 기능 완료 전 확인할 항목:
  - Issue의 완료 조건을 모두 충족했는가
  - 관련 화면 또는 API가 정상 동작하는가
  - 입력 검증과 에러 처리가 포함되었는가
  - 지도 마커 표시가 요구사항과 맞는가
  - 공유 방 참여자 기준의 접근 조건이 맞는가
  - 테스트 또는 수동 검증 결과를 남겼는가
- 테스트를 생략할 수 있는 경우:
  - 문서 수정만 있는 경우
  - 주석, README, 오타 수정처럼 런타임 동작에 영향이 없는 경우
  - 테스트 생략 시 완료 보고에 생략 이유를 명시한다.

## Documentation Rules

- 기획 문서 위치: `docs/PROJECT_PLAN.md`
- API 문서 위치: 미정. 백엔드 API가 만들어지면 Markdown 문서 또는 OpenAPI 형식으로 관리한다.
- 트러블슈팅 기록 위치: 미정. 프로젝트 내 `docs/troubleshooting/` 또는 Notion/블로그 중 하나로 정한다.
- 의사결정 기록 방식:
  - 데이터베이스, 지도 API, 인증 방식, 공유 정책처럼 중요한 결정은 결정 이유와 대안을 함께 기록한다.
  - GitHub Issue, Markdown 문서, Notion 중 하나에 링크 가능한 형태로 남긴다.
  - 개발 과정 회고와 문제 해결 과정은 문서화한다.

## AI Collaboration Rules

- AI 에이전트에게 맡길 수 있는 작업:
  - Issue 요구사항 분석
  - Vue/TypeScript 프론트엔드 구현
  - Node.js/Express 백엔드 구현
  - API, 컴포넌트, 테스트 코드 작성
  - 문서 초안 작성 및 정리
  - 버그 원인 분석과 수정안 제안
- 사람이 반드시 검토해야 하는 작업:
  - AI가 작성한 모든 코드의 최종 리뷰
  - 요구사항 충족 여부 판단
  - 데이터베이스와 지도 API 최종 선택
  - 보안, 개인정보, 공유 권한 정책
  - 배포 및 외부 서비스 비용이 발생하는 결정
- 구현 전 확인해야 할 요구사항:
  - 어떤 사용자 시나리오를 해결하는 기능인지
  - MVP 포함 범위인지 제외 범위인지
  - 관련 도메인 모델과 API 계약이 무엇인지
  - 완료 조건과 테스트 기준이 명확한지
  - 지도 API, 데이터베이스, 인증 등 미정 사항에 의존하는지
- 완료 보고 시 포함해야 할 내용:
  - 변경한 파일과 핵심 변경 사항
  - 구현한 요구사항
  - 실행한 테스트 또는 수동 검증 결과
  - 남은 이슈, 미정 사항, 리스크

## Do Not

- 하지 말아야 할 작업:
  - MVP 제외 범위 기능을 임의로 구현하지 않는다.
  - 공개 프로필, 팔로우, 고도화된 SNS 피드, 장소 추천 알고리즘, AI 자동 일기 생성, 결제, 네이티브 앱, 실시간 동시 편집을 임의로 추가하지 않는다.
  - 복잡한 권한 체계를 먼저 만들지 않는다. MVP에서는 공유 방 참여자가 공동 편집 가능하다는 단순 구조를 우선한다.
  - 요구사항 없이 기술 스택을 바꾸지 않는다.
- 수정하면 안 되는 파일:
  - 외부 생성 파일, 잠금 파일, 환경 파일은 변경 목적이 명확할 때만 수정한다.
  - 비밀 키, 토큰, 개인 환경 값은 저장소에 커밋하지 않는다.
- 임의로 결정하면 안 되는 사항:
  - 데이터베이스 최종 선택
  - 지도 API 최종 선택
  - 인증 방식과 공유 권한 정책
  - 배포 플랫폼과 유료 외부 서비스 사용

## Notes

- 이 프로젝트는 장소 중심 기록 경험과 공동 장소 관리 경험을 결합하는 것이 핵심이다.
- 구현 속도보다 요구사항 충족 여부, 유지보수성, 테스트 가능성, 보안 이슈를 우선한다.
- AI 에이전트가 작성한 코드는 반드시 사람이 리뷰하고 테스트한다.
- 주요 설계 결정과 트러블슈팅은 프로젝트 과정의 산출물로 남긴다.
