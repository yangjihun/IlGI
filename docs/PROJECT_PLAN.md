# 장소 기반 다이어리 서비스 기획서

## 1. 프로젝트 개요

본 프로젝트는 사용자가 방문한 장소를 중심으로 일상을 기록하고, 함께 가보고 싶은 장소를 다른 사람과 공동으로 관리할 수 있는 장소 기반 다이어리 서비스이다.

사용자는 오늘 방문한 장소에 대한 다이어리를 작성하고, 작성한 기록을 지도 위에서 한눈에 확인할 수 있다. 또한 초대 코드를 통해 공유 방을 만들거나 입장하여 다른 사용자와 장소 기반 다이어리를 공유할 수 있다.

서비스의 또 다른 핵심 기능은 가보고 싶은 장소 목록이다. 사용자는 원하는 카테고리를 직접 만들고, 장소를 카테고리별로 저장할 수 있다. 이 목록은 다른 사용자와 함께 추가, 수정, 삭제할 수 있어 단순 공유 링크가 아니라 공동 편집 가능한 장소 리스트로 동작한다.

최종적으로 이 서비스는 "함께 가보고 싶은 장소를 정하고, 실제로 다녀온 뒤 기록을 남기는 흐름"을 하나의 제품 경험으로 연결하는 것을 목표로 한다.

## 2. 기획 배경

작성자는 평소 계획하고 기록하는 것을 좋아하며, 기록을 다른 사람과 공유하고 나중에 회상하는 경험을 중요하게 생각한다. 현재는 별도의 다이어리 앱 대신 삼성 노트의 정해진 템플릿을 활용해 간단히 기록하고 있다.

기존 기록 방식은 간편하지만, 장소 중심의 기록을 시각적으로 모아보거나 다른 사람과 함께 관리하기에는 한계가 있다. 특히 작성자는 장소를 기준으로 다이어리를 작성하는 편이기 때문에, 기록한 장소들이 지도 위에 쌓이는 경험이 있으면 다이어리를 작성하는 과정 자체가 장소를 수집하는 느낌으로 확장될 수 있다.

또한 가보고 싶은 장소를 저장하는 기능은 네이버지도 등 기존 지도 서비스에도 존재하지만, 다른 사람과 함께 편집하기 어렵다. 링크를 공유할 수는 있어도 상대방이 같은 리스트를 함께 수정하는 공동 작업 경험은 제한적이다.

이 프로젝트는 이러한 불편함을 해결하기 위해 장소 기록과 장소 목록 공동 관리를 결합한 개인 토이 프로젝트로 기획되었다.

## 3. 프로젝트 목표

### 3.1 서비스 목표

- 방문한 장소를 기준으로 다이어리를 작성할 수 있다.
- 작성한 다이어리를 지도 위에서 시각적으로 확인할 수 있다.
- 초대 코드를 통해 다른 사람과 공유 방을 만들고 참여할 수 있다.
- 공유 방 안에서 장소 기반 다이어리를 함께 확인할 수 있다.
- 가보고 싶은 장소를 사용자가 만든 카테고리별로 저장할 수 있다.
- 가보고 싶은 장소 목록을 다른 사용자와 공동으로 추가, 수정, 삭제할 수 있다.

### 3.2 개인 성장 목표

- Vue와 Node.js 기반으로 프론트엔드와 백엔드를 직접 설계하고 구현한다.
- AI 에이전트인 Codex를 활용해 하네스 엔지니어링 방식의 개발을 실험한다.
- AI가 작성한 코드를 테스트하고 리뷰하면서 코드 독해 및 검증 역량을 기른다.
- 기능 단위로 GitHub Issue를 작성하고, 요구사항과 완료 조건을 명확히 정의하는 습관을 만든다.
- 개발 과정에서 발생한 의사결정, 문제 해결 과정, 트러블슈팅을 문서화한다.

## 4. 핵심 사용자 시나리오

### 4.1 방문 장소 다이어리 작성

사용자는 오늘 방문한 장소를 선택하거나 직접 입력한 뒤, 해당 장소에서 기억에 남는 일을 기록한다.

다이어리의 기본 구성은 다음과 같다.

- 날짜
- 장소
- 기억에 남는 것
- 추가 메모

작성된 다이어리는 지도 위의 장소 마커로 표시된다. 사용자는 지도를 보며 자신이 기록한 장소를 회상할 수 있다.

### 4.2 공유 방에서 다이어리 공유

사용자는 초대 코드를 통해 공유 방을 생성하거나 입장한다. 공유 방 안에서는 참여자들이 작성한 장소 기반 다이어리를 함께 확인할 수 있다.

초기 MVP에서는 복잡한 권한 체계보다 간단한 참여 구조를 우선한다. 공유 방에 참여한 사용자는 방 안의 다이어리와 장소 목록을 확인할 수 있고, 정책에 따라 공동 편집에 참여할 수 있다.

### 4.3 가보고 싶은 장소 저장

사용자는 가보고 싶은 장소를 저장하고, 직접 만든 카테고리에 따라 분류한다.

예시 카테고리는 다음과 같다.

- 맛집
- 카페
- 여행지
- 전시
- 산책 코스

저장된 장소는 나중에 실제 방문 장소를 정할 때 참고할 수 있다.

### 4.4 가보고 싶은 장소 목록 공동 관리

사용자는 가보고 싶은 장소 목록을 다른 사용자와 공유한다. 공유된 사용자는 단순 조회뿐 아니라 장소 추가, 수정, 삭제에 참여할 수 있다.

이를 통해 같이 갈 장소를 정하고, 실제 방문 후 다이어리 기록으로 이어지는 흐름을 제공한다.

## 7. 기술 스택

### 7.1 프론트엔드

- Vue
- TypeScript
- 지도 API 연동

Vue는 최근 학습한 기술을 실제 프로젝트에 적용하기 위해 선택한다. 초기에는 빠른 MVP 구현을 우선한다.

### 7.2 백엔드

- Node.js
- Express

토이 프로젝트의 특성상 Java 기반 서버보다 Node.js와 Express를 사용해 빠르게 MVP를 구현한다. 추후 기능이 확장되고 구조화 필요성이 커질 경우 NestJS로 확장하는 방안을 고려한다.

### 7.3 데이터베이스

초기 설계 단계에서 다음 기준으로 데이터베이스를 선택한다.

- 사용자, 장소, 다이어리, 공유 방, 카테고리 관계 표현이 쉬운가
- 위치 정보 저장 및 조회에 적합한가
- 토이 프로젝트 규모에서 운영 부담이 낮은가
- 추후 기능 확장에 대응할 수 있는가

위치 기반 데이터 처리가 중요해질 경우 PostgreSQL과 PostGIS 조합을 우선 검토한다. 초기 MVP에서는 구현 속도와 학습 비용을 함께 고려해 최종 선택한다.

### 7.4 협업 및 문서화 도구

- GitHub
- GitHub Issues
- Notion
- Markdown 문서
- Velog 또는 블로그 기록

개인 프로젝트이므로 Jira와 같은 별도 협업 도구는 초기에는 도입하지 않는다. 기능 단위 작업 관리는 GitHub Issues로 진행하고, 설계 문서와 회고는 Notion 또는 Markdown으로 관리한다.

## 8. 개발 방식

### 8.1 하네스 엔지니어링

이 프로젝트는 하네스 엔지니어링 방식을 실험한다.

큰 방향과 요구사항은 개발자가 정의하고, 정해진 틀 안에서 AI 에이전트가 코드를 작성하도록 한다. 개발자는 AI가 작성한 코드를 검토하고 테스트하며, 요구사항 충족 여부와 유지보수성을 판단하는 역할에 집중한다.

### 8.2 AI 에이전트

AI 에이전트로 Codex를 사용한다.

Codex를 선택한 이유는 단순히 코드를 빠르게 작성하기 위해서만이 아니다. 작은 규모의 토이 프로젝트에서 AI와 협업하는 방식을 실험하고, AI가 작성한 코드를 빠르고 정확하게 읽고 검증하는 능력을 기르기 위함이다.

### 8.3 작업 단위

각 기능은 GitHub Issue 단위로 쪼갠다. Issue에는 다음 내용을 포함한다.

- 기능 설명
- 요구사항
- 완료 조건
- 테스트 기준
- 관련 화면 또는 API

AI 에이전트에는 Issue 단위로 작업을 할당하고, 구현 후 개발자가 리뷰와 테스트를 수행한다.

## 9. 주요 도메인

### 9.1 사용자

서비스를 사용하는 주체이다. 다이어리를 작성하고, 장소 목록을 만들고, 공유 방에 참여한다.

### 9.2 다이어리

특정 날짜와 장소에 연결된 기록이다. 기억에 남는 내용, 작성자 정보를 포함한다.

### 9.3 장소

다이어리 작성 대상이 되거나 가보고 싶은 장소 목록에 저장되는 핵심 데이터이다. 지도 표시와 검색의 기준이 된다.

### 9.4 카테고리

가보고 싶은 장소를 분류하기 위한 사용자 정의 기준이다. 개인 또는 공유 목록 단위로 관리할 수 있다.

### 9.5 공유 방

여러 사용자가 함께 다이어리와 장소 목록을 공유하는 공간이다. 초대 코드를 통해 입장한다.

### 9.6 가보고 싶은 장소 목록

사용자 또는 공유 방 단위로 관리되는 장소 모음이다. 공동 CRUD 기능을 중심으로 한다.

## 12. 리스크 및 고려사항

### 12.1 지도 API 선택

지도 기반 서비스이므로 어떤 지도 API를 사용할지 결정해야 한다. 장소 검색, 마커 표시, 비용, 사용량 제한, 문서 품질을 기준으로 비교한다.

### 12.2 공유 기능의 복잡도

공유 방과 공동 편집은 권한, 데이터 정합성, 동시 수정 문제를 만들 수 있다. MVP에서는 복잡한 권한 체계보다 "공유 방 참여자는 공동 편집 가능"이라는 단순 구조로 시작한다.

### 12.3 AI 생성 코드 품질 관리

AI가 작성한 코드는 반드시 사람이 리뷰하고 테스트해야 한다. 구현 속도보다 요구사항 충족 여부, 유지보수성, 테스트 가능성, 보안 이슈를 우선 검토한다.

## 13. 성공 기준

- 사용자가 방문한 장소에 대한 다이어리를 작성할 수 있다.
- 작성한 다이어리가 지도 위에 표시된다.
- 사용자가 가보고 싶은 장소를 카테고리별로 저장할 수 있다.
- 초대 코드를 통해 공유 방을 만들고 입장할 수 있다.
- 공유 방 참여자가 장소 목록을 함께 추가, 수정, 삭제할 수 있다.
- 프로젝트 과정에서 주요 설계 결정과 트러블슈팅이 문서로 남는다.
- AI 에이전트가 작성한 코드에 대해 리뷰와 테스트를 수행하는 개발 흐름을 경험한다.

## 14. 결론

이 프로젝트는 단순한 다이어리 앱을 만드는 것을 넘어, 장소 중심의 기록 경험과 공동 장소 관리 경험을 결합하는 것을 목표로 한다.

사용자는 가보고 싶은 장소를 함께 정하고, 실제 방문 후 다이어리를 남기며, 시간이 지나 지도 위에 쌓인 기록을 통해 경험을 회상할 수 있다.

개발 측면에서는 Codex를 활용한 하네스 엔지니어링을 실험하며, AI 시대의 개발자가 갖춰야 할 설계, 검증, 코드 리뷰 역량을 실제 프로젝트를 통해 훈련하는 데 의미가 있다.
